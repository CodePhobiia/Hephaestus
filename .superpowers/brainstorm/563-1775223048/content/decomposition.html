This spec describes 8+ independent subsystems. Each needs its own design → plan → build cycle.

A

Start with P1: Storage & Data Model

Vault, Source, Page, Claim, Link, Workbook — the core schema everything builds on. Extends existing RunStore (Postgres) and ResearchArtifactStore (SQLite). Defines the data contracts before anything else touches them.

Pros

  • Foundation-first, no rework
  • Can validate schema with tests before building on top

Cons

  • Abstract until ingestion/compilation exist
B

Start with P1+P2 together: Storage + Ingestion

Design the data model AND the ingestion pipeline as one unit. This way the schema is immediately exercised by real data flowing through it — PDFs, URLs, markdown, prior run artifacts.

Pros

  • Schema validated by real use immediately
  • Ingestion extends existing PerplexityClient + ingestion.py

Cons

  • Larger first spec, but both are well-bounded
C

Start with a vertical slice: Storage + Ingestion + Basic Compilation

Design a minimal end-to-end path: create vault, ingest sources, compile basic concept/source pages. Proves the architecture works before going wide. Defers provenance, linting, agent teams, and Genesis integration.

Pros

  • Usable artifact after first sub-project
  • Validates the full data flow early

Cons

  • Largest first spec — 3 subsystems
  • Risk of premature compilation design
